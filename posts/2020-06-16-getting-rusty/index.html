Getting Rusty

Jun 16, 2020
3 min read

About 2 weeks ago, I read an article about the programming language Rust has won the most loved language in Stackoverflow survery for 2020. For 5 years in a row.

It was about 5 years ago when I looked into Go (Golang) and at that time I read a bunch of comparisions between Go and Rust. And somehow, I didn’t end up trying it. Go was great and ended up building many tools with it. I love its simplicity and speed.

In the last 2 years, I used Go quite a bit and ended up building a pretty extensive system with several microservices written in Go. It’s performing really well, I must say. But I started to come up against some of the limitations of the language, and after debugging some data race situations I started to think that maybe there’s a better way to do things.

I heard about the claims about Rust that it’s much safer when handling memory and can catch almost all data race situations at compile time. And that article recently was the final nudge for me to take a closer look.

I started studying up on it and I went through a couple tutorial, bunch of Youtube videos, etc and by this time I get a pretty good idea about it.

The things I like so far:

  • Error handling: OMG, once I understood how it works, it’s so much better than Go’s tedious if err!= nil {...} blocks.
  • The type system: Especially the enums look really useful in many cases I worked on.
  • Implementation of functional programming concepts: I recently did a study up on functional programming and I was able to utilize it quite a bit when working with Javascript. But Go had very little of it implemented besides functions that can be passed as parameters. Rust seems to strike a good balance between a pure functional language and a poorly functional language, by allowing side effects and mutations but making them very clear.
  • Generics: I missed this in Go. I think a statically typed language should have it because otherwise things get rigid. Go’s workaround with the interface{} types can get quickly unwieldy (and error prone - causing runtime crashes).
  • Macros: Many times when writing my code in Go, I was wishing to avoid the repetition of code but in many cases I couldn’t because they would have required compile time changes (and Generics). I haven’t learned this part yet, but I’m looking forward to it.

So far, I haven’t found much of a drawback to it. Except for a bit steeper learning curve - and somewhat disorganized learning materials. I think I managed to understand the principles of borrowing

I need to find a good entry project to try it out in production too. But so far, I’m excited about it.
