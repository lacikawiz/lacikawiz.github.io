HostWay - probably the most outdated web host provider

Jun 18, 2020
3 min read

Today, I migrated a WordPress site from the staging area under my own hosting to my client’s hosting (HostWay).

The old site was running on some PHP 5-something. Their control panel is a home made one with hardly any CSS, and it didn’t give me an option to change the PHP version. It’s a pretty standard feature on most hostings. So, I contacted tech support via chat and asked them to upgrade it. They upgraded it to version 7.0. I asked them if they could be upgrade to a bit highger version of PHP. They said “no” and quickly added “that there’s no ETA when it will be possible” - without me asking. I guess, it was a pretty common question. The guy ensured me that the latest version of WordPress will run on this.

So, I migrated the files for the site. Set up a new MySQL DB. And well, PHPMyAdmin didn’t work. When I click on the link, it came back with some odd errors. So back to tech support… The guy said he sees the error but can’t fix it, needs to escalate it. Since it was the middle of the night in the US, I knew it won’t be fixed for several hours at least.

But I had to get the site up and running before visitors start to come back. I already took down the old one and it would be a hassle to put it back. Switch the PHP to the old version, etc. So, I steeled myself and faced doing the SQL import the hard way – via command line.

Fortunately, they had SSH access. So I could go in, after some fumbling around with the command line switches I was able to get connected to the database and imported the SQL file. But I was getting some error message about unsupported “collation”. The file I had contained many “ut8_mb4…” things. I thought maybe it was just some unusual thing, so I switched out all these to simple utf8. So the import went through, and the website came up. Yay!!

Well, the celebration didn’t last long. I logged into the back-end. And was greeted by two dire warnings:

  • WordPress has detected that your site is running on an insecure version of PHP.
  • Pods 2.7.20.1 requires a minimum of MySQL 5.1+ to function. You are currently running MySQL 5.0.51

And the last one is particularly sucks because I used PODS extensively on this site and now the only way I can update the content is with directly editing the database tables. Gosh!

A little digging around: PHP 7.0 was released in Dec 2015, and went out of support in Dec 2018. Not so bad.

However, MySQL 5.0 was release in Mar 2005, and 5.1 was release in Nov 2008. So it’s ancient! That’s why it couldn’t recognize the encodings. They probably didn’t even exist when that version was released.

I will give a try if I can get tech support to upgrade the MySQL. But I’m not holding my breath on that. Everything about this company suggest that they didn’t move much since 2010.

One thing is pretty nice though. The tech support is easily reachable and they do understand what I wrote. They don’t seem to be using some underpaid, outsourced tech support. And they also fixed the glitch with PHPMyAdmin within a few hours after the sun came up in the US.

BTW, the site runs a lot slower than even on my budget 1AND1 hosting.

Now, I’m trying to sell my client the idea of switching the hosting. Without PODS working, the site maintenance is pretty tough.

I don’t know how these dinosaurs survive. Maybe they keep things from changing in order to cater to a small number of companies with old sites.
